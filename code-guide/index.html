1. OMC（oh-my-claudecode）— マルチエージェント拡張

OMCとは？

oh-my-claudecode（OMC）は Claude Code に乗せる多エージェントオーケストレーション層です。
専門エージェントへの委譲・スキルの自動発動・Tier-0ワークフローにより、複雑なタスクを正確かつ効率よく完了します。
~/.claude/CLAUDE.md に設定済み。特別なインストール不要で有効です。

Tier-0 ワークフロー — キーワードで自動発動
以下のキーワードをチャットに入力するだけでワークフローが起動します。スラッシュコマンド不要。

計画→実装→検証を完全自律で完走。途中で詰まっても自己修正して前進する。

2. 開発の黄金ワークフロー

① 計画

まず「何を作るか」を合意する

「新しい機能を追加したい」「バグを直したいが影響範囲が不明」

いきなりコードを書かせず、要件・リスク・依存関係を洗い出してから始める。後戻りが激減する。

/dev-routine /plan /orchestrate
② 実装

テストから書いて品質を最初から担保

「バグが後から見つかる」「型エラーが大量に出た」「UIを新しく作る」

テストを先に書くことで仕様の曖昧さを排除し、ビルドエラーはAIに自動修正させる。

/tdd /ui-skills /build-fix /feature-dev
③ 検証・コミット

コミット前に穴を全部塞ぐ

「プッシュする前に確認したい」「セキュリティが心配」「テストが足りているか不安」

セキュリティ・カバレッジ・ビルドを一括チェックして、安心してコミット・PRを出す。

/code-review /test-coverage /verify /commit /commit-push-pr

3. よくある困りごと別：使うべきコマンド

「コードが肥大化してきた、整理したい」

技術的負債の解消

  • 1
    /refactor-clean — 使っていない関数・変数・import を knip で検出して安全に削除
  • 2
    /update-codemaps — 整理後にアーキテクチャ図を再生成して全体像を把握
「READMEが古い、新メンバーが迷っている」

ドキュメントの同期

  • 1
    /update-docs — package.json の scripts と .env.example からREADME・RUNBOOKを自動更新
  • 2
    /update-codemaps — ディレクトリ構造・依存関係の地図を最新化
「同じミスをAIが繰り返す、学習させたい」

AIを自分好みに育てる

  • 1
    /checkpoint — 大きな変更の前に git ベースのセーブポイント作成
  • 2
    /learn — 「今回発見したパターン」をスキルファイルとして保存
  • 3
    /revise-claude-md — セッションの気づきを CLAUDE.md に書き戻す（次回から自動参照）
「ユーザーフローが壊れていないか確認したい」

E2Eテストで品質保証

  • 1
    /e2e ログイン〜購入フローのテスト — Playwright でシナリオを自動生成・実行
  • 2
    /verify pre-pr — E2E含む全テストをまとめて最終確認
「会話が長くなってきて応答が遅い・ズレる」

コンテキスト管理

  • 1
    /cost — 現在のトークン使用量を確認してどのくらい使ったか把握
  • 2
    /compact — コンテキストを圧縮して速度・精度を回復
  • 3
    /context — 現在の会話コンテキスト状態を確認
「PRを誰かにレビューしてもらう前に確認したい」

PR提出前チェック

  • 1
    /code-review — 未コミット変更を CRITICAL/HIGH/MEDIUM で重要度分類
  • 2
    /verify pre-pr — ビルド・型・テスト・Lint・secrets を全部確認
  • 3
    /review-pr — PRナンバーを指定して包括的なPRレビューを実施

5. 使いこなしのコツ

「とりあえずコードを書かせる」をやめる

まず /plan で要件と影響範囲を合意してから実装を始めると、後戻りが激減する。3ステップ以上の作業なら必ず計画から。

大きな変更の前に必ず /checkpoint

AIにリファクタや大量のファイル書き換えをさせる前に /checkpoint create 名前 を実行。git コミットとは別の「AIとの対話のセーブポイント」として機能する。

コマンドをチェーンする

/plan の計画を承認したら「この計画で /tdd を実行して」と続けるだけ。コマンド間のハンドオフはAIが処理する。完了後に /verify pre-pr。

/revise-claude-md でフィードバックを蓄積

「AIに3回同じことを説明した」と気づいたら /revise-claude-md でセッションの発見を CLAUDE.md に書き戻す。次回から自動でその知識が使われる。

会話が長引いたら /cost → /compact

/cost でトークン消費を確認して上限が近づいたら /compact で圧縮。応答速度と精度が改善する。重要な節目（機能完成後など）が圧縮タイミングの目安。

コミットを自動化する

手動で git add + コミットメッセージを書く代わりに /commit で変更を解析して適切なメッセージを自動生成。さらに /commit-push-pr でPR作成まで一発。

6. ヘッドレスモード（自律実行）

対話なしでClaudeを自律実行させるモードです。「計画ループ」で詰まることなく、完了まで走り切ります。

基本構文

claude -p "指示" --allowedTools "Edit,Read,Bash,Write,Grep"

活用レシピ

# テスト全修正（放置して完了を待つ）
claude -p "Fix all failing tests in tests/ directory and ensure pytest passes" \
  --allowedTools "Edit,Read,Bash,Write,Grep"

# 特定機能の実装（計画なし・即実装）
claude -p "実装内容を説明。計画せずすぐにコードを書いて、テストを通してコミットして" \
  --allowedTools "Edit,Read,Bash,Write,Grep"

# コードレビューをサイレント実行
claude -p "Review src/ for security issues and output a report" \
  --allowedTools "Read,Grep,Glob" --output-format json

通常モードとの使い分け

状況推奨モード
仕様が曖昧・相談しながら進めたい通常の対話モード
タスクが明確・放置して完了を待ちたいヘッドレスモード (-p)
CIや定期バッチで自動実行したいヘッドレスモード (-p)
複数プロジェクトを並列処理したいヘッドレスモード × 複数ターミナル
