マツココライブ（MCCM）の年次報告会向けドキュメント・データ管理プロジェクト。配信実績の分析、メーカー向けレポート、月次進捗報告など、複数のステークホルダー向けドキュメントを統合管理しています。

## よく使うタスク

### 1. 配信数値の分析と集約
**スキル**: `delivery-analysis`（ライブ配信のデータを分析し、インサイトと改善提案を含む分析レポート生成）

**用途**: CSV配信数値データから月別・商品別・メーカー別の傾向を抽出し、レポートを作成

```bash
# 例：配信数値CSVを読み込んで分析レポート生成
- 【マツココライブ】配信数値まとめ - 数値まとめ.csv を分析
- 期間別・カテゴリ別の成長率、KPIを抽出
- 分析結果を新規レポートとして保存（例: 2025年度配信分析_${DATE}.md）
```

### 2. 年次報告会資料の作成・更新
**スキル**: `pitch-deck`（クライアント向けのピッチデック・提案資料の構成と文章を作成）

**用途**: 年度統括報告会用PowerPoint資料の構成案、スライド文案を生成

```bash
# 例：メーカー実績をまとめた報告資料を作成
- 05_協賛配信関連/メーカー提出実績レポート/2024, 2025.1-6, 2025.7-12 から主要データを抽出
- 年度別の成長推移、メーカー別の実績を集計
- 報告会向けスライド構成案・説得力のある文案を生成
```

### 3. 戦略・収益シミュレーションの企画
**スキル**: `strategy-plan`（クライアント向けの戦略プラン・企画書・収益シミュレーションを作成）

**用途**: 次年度の配信戦略、事業計画、売上見通しの企画

## ドキュメント管理のルール

### ファイル命名規則

**PowerPoint資料**
- 形式: `【MCCM様】月次コンテンツ分科会_YYYYMMDD.pptx`
- メーカー向けレポート: `ライブコマースレポート_ブランド名.pptx` または `.pdf`

**分析・レポート**
- 形式: `${テーマ}_${日付}.md` または `${テーマ}_${日付}.xlsx`
- 例: `配信分析_20250326.md`, `メーカー実績集計_2025年度.xlsx`

**重要**: 作成者の修正版は「_修正」や「_佐藤修正」などを末尾に付加

### バージョン管理

- 作業中ドラフト: 「_ドラフト」を末尾に付加
- 確定版は新規ファイルとして保存し、古い版に「_old」を付加（上書きしない）
- PDFは参照用複製のため .pptx と並行管理

### 期間分類の原則

メーカー実績レポートは以下の期間で分類：
- **2024/**: 2024年度（2024年4月〜2025年3月）のメーカーレポート
- **2025.1-6/**: 2025年1月〜6月期のメーカーレポート
- **2025.7-12/**: 2025年7月〜12月期のメーカーレポート

月次分科会資料は **過去の振り返り資料** に統一保存し、日付の新しい順で並べ替え

## データの扱い方

### CSV配信数値データ

**ファイル**: 【マツココライブ】配信数値まとめ - 数値まとめ.csv

分析時には以下を抽出：
- 配信日時、商品名、売上、視聴数などの基本指標
- メーカー別・カテゴリ別・期間別の集計
- 前期比成長率、トレンド分析

新規データ追加時は、既存CSVに行を追加（新規ファイル作成はしない）

### Firework公式指標定義（重要・分析資料の参考情報として必ず記載）

Fireworkダッシュボード／CSVに出てくる「エンゲージ率」「アクティブ率」は公式ドキュメントに明記がない独自指標。ライブ配信分析レポート・提案書では**初出時に必ず以下の定義を注釈として入れる**。

| 指標 | 正式定義（Firework公式） |
|---|---|
| **Visitors With Engaged Views**（エンゲージ視聴者数） | ライブ配信を最大化 or Storyblockで3秒以上視聴、またはすぐにボタンクリックなどのエンゲージメントをしたユニークユーザー数 |
| **Visitors With Active Views**（アクティブ視聴者数） | ライブ配信内で、ボタンクリックなどのエンゲージメントをしたユーザー数 |
| エンゲージ率 | エンゲージ視聴者数 ÷ active視聴者数 |
| アクティブ率 | アクティブ視聴者数 ÷ active視聴者数 |

**注意点**:
- CTR + コメント率 + いいね率の単純合計より値が大きくなる配信がある（最大化視聴・Storyblock 3秒視聴などが加算されるため）
- CSVでは `active`／`active_repeat`／`replay` の3セクションそれぞれに同じ指標が存在
- 出典：Firework担当者経由ヒアリング（2026-04-20 確定）
- グローバル参照：`~/.claude/projects/-Users-issy--------02-Claude-Code-02----03--------MCCM/memory/reference_firework_engagement_metrics.md`

## 作業フロー（重要）

以下の順で進める：

1. **タスク確認**: 作業内容・出力形式を明確にする
2. **データ収集**: 必要なCSV、既存資料を読み込み・確認
3. **分析/企画**: スキルを使用して内容を生成
4. **確認**: 生成物の妥当性、数値の正確さを検証
5. **保存**: 命名規則に従い、新規ファイルとして保存
6. **報告**: 作成内容、ファイル保存先を報告

ドラフト段階でユーザーの確認を求めることを推奨。修正指示を受けてから最終版を確定保存。

## 禁止事項

- ファイルの上書き保存 → 新規ファイルとして別名で保存
- 既存資料の削除 → 不要な資料は「_old」を付加して保存継続
- 外部ツール連携の自動実行 → 実行前にユーザーに内容を見せて承認を得る

## 競合数値データの取り扱い（重要）

- **競合4社の `active_repeat` データは `04_過去資料/参考資料/競合他社数値まとめ/` に存在**（資生堂・アットコスメ・ロート製薬はxlsx内シート、KANEBOはCSV）。当初「非公開」と判断しがちだが必ず先にここを確認すること
- **比率スケールが混在**：マツキヨCSVは 0-100（%）、競合xlsxは 0-1（小数）。集計時は必ず自動判定（`max ≤ 1.5 → *100`）で揃える。詳細：`~/.claude/skills/learned/csv-excel-ratio-scale-detection.md`
- **Firework公式に `active_repeat` 指標は存在しない**（マツキヨ独自実装）。定義は "同一配信内再入室" か "過去配信視聴経験あり" か不明 → Firework担当者経由で確認が必要
- **openpyxl 実行**：`/usr/bin/python3` 経由で `~/Library/Python/3.9/lib/python/site-packages` のパッケージが使える（`python3` 直叩きは `.browser-use-env` の方を見て `ModuleNotFoundError` になる）

## ドキュメント更新ルール

- **HTML編集時は対応するmdを必ず同時に同じ構成で更新**（片方だけ更新しない）。基本フロー：**mdを先に直してからhtml**。セクション追加・削除・並び替え・タイトル変更・内容修正すべてを両ファイルで同期
- 競合調査書の正本：`02_データ/競合/競合調査・KPI設計_提案書.md` ↔ `02_データ/競合/競合調査・KPI設計_提案書_20260415.html`
- 長いHTMLのセクション並び替え／削除／リナンバーはPythonで一括処理。詳細：`~/.claude/skills/learned/html-section-restructure-pattern.md`

### 🔁 HTML編集ルーティーン（MANDATORY）

**cp方式は禁止**（公開版に含まれる SSO 2行 `<script src="auth.js">` / `FireworkSSO.protect()` が消えるため）。以下の手順で必ず進める：

1. **骨子版HTMLを編集**（正本）
   - 統合版: `01_年次報告会/提案骨子/01_提案書/提案に向けた情報まとめ_統合版_20260414.html`
   - 分科会用: `01_年次報告会/提案骨子/01_提案書/分科会用資料_20260414.html`

2. **公開版との差分確認**
   ```bash
   diff "01_年次報告会/提案骨子/01_提案書/提案に向けた情報まとめ_統合版_20260414.html" "mccm-proposal/index.html"
   ```
   → SSO2行とコンテンツ差分が両方見える。SSO2行は公開版のみに存在するため必ず残す。

3. **公開版にも同じ変更を Edit ツールで反映**（cp不可）
   - 骨子版で変更した箇所と同じ `old_string → new_string` を公開版にも適用
   - 3箇所以上変わる場合は一つずつ Edit で順に差し替える

4. **両ファイルの Last Updated を実時刻で更新**（必ず両方）
   ```bash
   date '+%Y年%-m月%-d日 %H:%M'
   ```
   → 出力結果を両ファイルの `<div class="value">` に貼る。思い込み・キリのいい時刻は禁止。

5. **git add → commit → push**
   ```bash
   git add "mccm-proposal/index.html" "01_年次報告会/提案骨子/01_提案書/提案に向けた情報まとめ_統合版_20260414.html"
   git commit -m "mccm: {変更内容}"
   git push
   ```
