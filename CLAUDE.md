## ドキュメント管理のルール

### ファイル命名規則

**PowerPoint資料**
- 形式: `【MCCM様】月次コンテンツ分科会_YYYYMMDD.pptx`
- メーカー向けレポート: `ライブコマースレポート_ブランド名.pptx` または `.pdf`

**分析・レポート**
- 形式: `${テーマ}_${日付}.md` または `${テーマ}_${日付}.xlsx`
- 例: `配信分析_20250326.md`, `メーカー実績集計_2025年度.xlsx`

**重要**: 作成者の修正版は「_修正」や「_佐藤修正」などを末尾に付加

### バージョン管理

- 作業中ドラフト: 「_ドラフト」を末尾に付加
- 確定版は新規ファイルとして保存し、古い版に「_old」を付加（上書きしない）
- PDFは参照用複製のため .pptx と並行管理

### 期間分類の原則

メーカー実績レポートは以下の期間で分類：
- **2024/**: 2024年度（2024年4月〜2025年3月）のメーカーレポート
- **2025.1-6/**: 2025年1月〜6月期のメーカーレポート
- **2025.7-12/**: 2025年7月〜12月期のメーカーレポート

月次分科会資料は **過去の振り返り資料** に統一保存し、日付の新しい順で並べ替え

## データの扱い方

### CSV配信数値データ

**ファイル**: 【マツココライブ】配信数値まとめ - 数値まとめ.csv

### Firework公式指標定義（重要・分析資料の参考情報として必ず記載）

Fireworkダッシュボード／CSVに出てくる「エンゲージ率」「アクティブ率」は公式ドキュメントに明記がない独自指標。ライブ配信分析レポート・提案書では**初出時に必ず以下の定義を注釈として入れる**。

| 指標 | 正式定義（Firework公式） |
|---|---|
| **Visitors With Engaged Views**（エンゲージ視聴者数） | ライブ配信を最大化 or Storyblockで3秒以上視聴、またはすぐにボタンクリックなどのエンゲージメントをしたユニークユーザー数 |
| **Visitors With Active Views**（アクティブ視聴者数） | ライブ配信内で、ボタンクリックなどのエンゲージメントをしたユーザー数 |
| エンゲージ率 | エンゲージ視聴者数 ÷ active視聴者数 |
| アクティブ率 | アクティブ視聴者数 ÷ active視聴者数 |

**注意点**:
- CTR + コメント率 + いいね率の単純合計より値が大きくなる配信がある（最大化視聴・Storyblock 3秒視聴などが加算されるため）
- CSVでは `active`／`active_repeat`／`replay` の3セクションそれぞれに同じ指標が存在
- 出典：Firework担当者経由ヒアリング（2026-04-20 確定）
- グローバル参照：`~/.claude/projects/-Users-issy--------02-Claude-Code-02----03--------MCCM/memory/reference_firework_engagement_metrics.md`

### Slides生成の確定ノウハウ（2026-04-27）

- **タイトル黒帯座標は絶対不変**: `x=9, y=20, w=671, h=45`（18pt bold, 白文字, START, MIDDLE）
- **配色は黒ベース**（`C_DARK=#1A1A2E`）。紺色（`#0F3460`）は使わない
- **HTML→スライド変換時は80%カット原則**：HTML側の充実セクションをそのままスライドに入れず、要素数を1/5に絞る。1スライドカードは「上部バッジ＋区切り線＋狙い文」の3要素以内が黄金値。詳細：`~/.claude/skills/learned/html-to-slide-80percent-cut-rule.md`
- **🔴 強調色は `#FA006D`（マゼンタ）で絶対統一**（2026-04-28 確定・MCCMリポジトリ全ファイルで `#E94560` を `#FA006D` に一括置換済み）
  - 対象: テキストインライン強調・KPI数値・タイムライン重要マイルストーン・カードのアクセント色・HTML CSS変数すべて
  - GAS: `02_helpers.gs` の `C_ACCENT` がこの色を持つ。新規スライド・新規関数では**必ず `C_ACCENT` 経由で参照**しハードコードしない
  - HTML/CSS: `#fa006d` を直書き（`color:`, `border-color:`, `background:`, `border-left:` 等すべて）
  - **🚨 セルフチェック必須**: 新規HTML・スライド・コードを書いた直後に `grep -ni "e94560" <file>` を実行し0件であることを確認。1件でもあれば即修正
  - **薄ピンク背景 `#fff5f7` は別色**（背景色用）として保持してOK。混同しない
  - 詳細メモ: `~/.claude/projects/-Users-issy-s-MCCM/memory/feedback_mccm_accent_color_fa006d.md`
- **🔡 ※注釈は8pt統一**（2026-04-28 確定）。「※ 〜」で始まる注釈テキストのフォントサイズは必ず `size: 8` で統一する。9pt・10ptは使わない。色は `C_GRAY`、配置は START。本文・小見出しは従来通り
- **🧱 オブジェクト+テキストはハイブリッド方針**（2026-04-28 確定）
  - **基本**: `rectTxt(slide, text, x, y, w, h, fill, opts, stroke, strokeW)` で**1要素化**（矩形に直接テキスト埋め込み・Slides UIで1クリック選択可）
  - **対象**: タイトル黒帯・カード本体ラベル・注釈BOX・メッセージBOX・章タイトル・単純な見出しBOX等
  - **例外（`rect` + `txt` のレイヤー継続OK）**:
    1. 1要素内で複数フォントサイズ・色を切り替える場合（例: KPIカードの「200万」太字+「人超」小文字）
    2. インライン部分強調が必要な場合（例: 結論メッセージで「年間1,440,000人」だけ #FA006D）
    3. 上ボーダー薄帯+ラベルなど装飾と本文を分けたい場合
  - 無理して力技で1要素にする必要なし。ハイブリッドで運用
- **タイトル黒帯はマスター由来の装飾**で、`getPlaceholder` では取得不可。`txt()` で同座標に白文字を新規挿入
- **Noto Sans JP Bold文字幅係数**（Playwright実測）: 半角数字=0.71, 全角=1.0, カンマ=0.27, スラッシュ=0.52
- **GAS APIではドロップシャドウは取得・設定不可** → 生成後にユーザーが UI から手動で「書式設定 → 影」を一括適用
- **ユーザー手動修正版を完全踏襲する**: AI汎用ヘルパー禁止、各オブジェクトを実測絶対座標で再現
- **HTMLとスライドの色配置は意図を分けてOK**：HTMLは直感色（緑=良/赤=危険）、スライドは役割色（マゼンタ=強調/紺=冷静な課題提起/緑=未来への提案）として整合性のある別運用にできる。例：振り返りセクションのスライド版（`14_section_review.gs`）でこのパターンを確立
- 詳細: `~/.claude/skills/learned/gas-slides-noto-sans-jp-text-width.md` / `user-edited-layout-faithful-reproduction.md` / `playwright-font-measurement-without-bash.md`

## 作業フロー（重要）

以下の順で進める：

1. **タスク確認**: 作業内容・出力形式を明確にする
2. **データ収集**: 必要なCSV、既存資料を読み込み・確認
3. **分析/企画**: スキルを使用して内容を生成
4. **確認**: 生成物の妥当性、数値の正確さを検証
5. **保存**: 命名規則に従い、新規ファイルとして保存
6. **報告**: 作成内容、ファイル保存先を報告

ドラフト段階でユーザーの確認を求めることを推奨。修正指示を受けてから最終版を確定保存。

## エグゼクティブ向け資料の必須フロー（MANDATORY）

**MCCMの年次報告会・四半期報告会資料、執行役員向け資料、経営層が見る提案書**を作成・修正する場合、ユーザー（Issy）へ提案する**前に必ずセルフレビュー**を実施する。エグゼクティブはロジックの穴を即座に見抜く。1箇所でも論理破綻すると資料全体の信頼性が崩れるため、提案前の自己評価が最も低コストで効果が高い。

### 3ステップ・セルフレビュー

#### Step 1: 因果関係の妥当性チェック
- 「A→B」「Aの原因はB」と書いた箇所を**すべて**洗い出す
- 相関と因果を混同していないか / 裏付けデータがあるか / 第三の変数を見落としていないか
- 🟢強い因果（裏付けあり） / 🟡弱い相関（曖昧表現に書き換え） / 🔴論理飛躍（削除）

#### Step 2: 突っ込まれリスク評価
エグゼクティブ目線で**最低3つ**の想定質問：
- 「本当にそうか？」「他に原因はないのか？」「根拠は？」
- 「競合はどうか？」「それで何が変わる？」「コスト・リスクは？」
- 即答できないものがあれば書き直し or 断定回避

#### Step 3: 表現の鋭さチェック
- 断定表現が強すぎないか
- 自社（Firework）の落ち度を不必要に強調していないか
- 論理の段差はないか（A→B→Cで Bが抜けていないか）

### 提案時の伝え方

セルフレビューで懸念が出た場合、提案時にユーザーへ**リスクと書き換え案を併記**して提示：

```
〔提案内容〕

セルフレビューで以下が気になりました：
- A→Bの因果関係が弱いかも → 突っ込まれリスクあり
書き換え案：案1（曖昧化） / 案2（削除）
```

→ ユーザーに「リスクと選択肢」を見せた上で判断してもらう

## 競合数値データの取り扱い（重要）

- **競合4社の `active_repeat` データは `04_過去資料/参考資料/競合他社数値まとめ/` に存在**（資生堂・アットコスメ・ロート製薬はxlsx内シート、KANEBOはCSV）。当初「非公開」と判断しがちだが必ず先にここを確認すること
- **比率スケールが混在**：マツキヨCSVは 0-100（%）、競合xlsxは 0-1（小数）。集計時は必ず自動判定（`max ≤ 1.5 → *100`）で揃える。詳細：`~/.claude/skills/learned/csv-excel-ratio-scale-detection.md`
- **Firework公式に `active_repeat` 指標は存在しない**（マツキヨ独自実装）。定義は "同一配信内再入室" か "過去配信視聴経験あり" か不明 → Firework担当者経由で確認が必要
- **openpyxl 実行**：`/usr/bin/python3` 経由で `~/Library/Python/3.9/lib/python/site-packages` のパッケージが使える（`python3` 直叩きは `.browser-use-env` の方を見て `ModuleNotFoundError` になる）

## よくある落とし穴

- **git status で "D + ??" を見たらrenameを疑う**：フォルダリネーム時は `git commit` で自動的にrename判定される（100%一致時）。削除と誤判断して手動復元しないこと
- **HTML編集時のSSO 2行**：`mccm-proposal/` 配下の index.html には `auth.js` 読み込み2行があり、cp方式で上書きすると消える → 必ず Edit ツールで差分反映する（詳細は「🔁 HTML編集ルーティーン」参照）
- **GAS Slides の `getSlides().length` と Slides本文の「P.X / Y」表示は別物**：GASログの「N枚→N+1枚」は実際のスライド総数だが、ユーザーが見ている「P53」等はスライド本文に書き込まれた静的テキストのフッター。新規追加スライドが見つからないと感じたら、まずスライド一覧（Cmd+Option+1）を末尾までスクロール。それでも探せない時はタイトル文字列で検索する関数を作って探す
- **視聴者数の直近値は 5,084人（2026.1-3 自社集計平均）で統一**。古い値「6,192人（12-2月の分科会資料ベース）」は誤記として削除済み。再追加しないこと
- **Edit時の文字化け注意**：全角括弧「（）」・カギ括弧「」・中点「・」などは必ず原文通りに old_string にコピー。半角「()」に勝手に変換すると `String to replace not found` で失敗する
- **Edit が3回連続で失敗したら Write に切り替える、複数箇所×複数ファイルの文言修正は最初からPython一括置換**：MD/HTMLで日本語括弧（全角）と英文括弧（半角）が混在するファイルは、Edit の `old_string` 完全一致が取れないことが多い。3回失敗したらファイル全文を Read → Write で全置換する方が速い。**さらに、複数ファイルにまたがる文言修正（例：「お得→おトク」を5箇所×2ファイル）は Python ヒアドキュメントで一括置換した方が手戻りが少ない**
- **リストでポジ＋ネガ事実を独立2項目で並べると誤読リスク**：「視聴者数+ / 視聴分数-」のように振り返りセクションでポジティブ事実とネガティブ事実を独立並列で書くと、前半がHighlightと混同される。「〜する一方、〜」で1メッセージに統合する
